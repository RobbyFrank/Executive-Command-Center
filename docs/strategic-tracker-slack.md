## Team → Import from Slack

Calls `users.list` with the bot token (`SLACK_BOT_USER_OAUTH_TOKEN`), then enriches **join dates** with `users.profile.get` (bot scope **`users.profile:read`**) when `users.list` omitted `profile.start_date`—Slack often returns a slimmer profile on list than on profile get. Join dates map from Slack Atlas **`start_date`** and ISO `YYYY-MM-DD` values in custom profile fields.

Then `team.billableInfo` runs with a **separate user token** (`SLACK_BILLING_USER_TOKEN`, xoxp- with user scope `admin`) because Slack rejects bot tokens for `team.billableInfo` (`not_allowed_token_type`).

The import list includes members whose billing status is **Active** (`billing_active: true` in `billable_info`) **or** **Active guest** (multi- or single-channel guests: `is_restricted` / `is_ultra_restricted`, excluding pending invites via `is_invited_user`). The server omits bots, Slackbot, and **deactivated** users (`deleted`) before that filter. The dialog shows a **Billing** column (Active vs Active guest) and skips members whose Slack user ID is already stored on a person row; profile images use Blob (`BLOB_READ_WRITE_TOKEN`).

Each newly imported row is then enriched from **that person's own Slack message history** (see [Message-based enrichment](#message-based-enrichment-role--department--join-date-fallback) below) to backfill **Role**, **Department**, and a fallback **Join Date** before the dialog closes. The enrichment is best-effort: if `search.messages` is not authorized, the row is still created with name/email/avatar/profile join date.

**New hire onboarding** (MPIM welcome detection, DM context for the pilot recommender, posting assignment messages) uses the same **user** token for `conversations.history` / MPIM membership as other flows; see [onboarding.md](onboarding.md).

## Team → Refresh all from Slack

Toolbar next to Import walks every roster row with a **`slackHandle`** and calls **`refreshPersonFromSlack`** once per person (same logic as the row **Refresh from Slack** menu: `users.info` for name/email/avatar, **`users.profile.get`** for join date when available, profile photo to Blob when configured). The UI runs these **sequentially** with a loading toast showing **progress (n / total)** and the **current name**; each refresh is its own server round trip. A bulk helper `refreshAllFromSlack` also exists on the server without per-step progress.

As part of each per-person refresh, the server also runs the same [message-based enrichment](#message-based-enrichment-role--department--join-date-fallback) as import—**only when** the roster row still has a blank **Role** or blank **Join Date** after profile data was applied—so existing values are never overwritten by the AI guess.

### Message-based enrichment (Role / Department / Join Date fallback)

Both **Import from Slack** and **Refresh all from Slack** augment their profile-based writes with Slack `search.messages` (user OAuth token, User scope **`search:read`**), implemented in `src/lib/slack/user-messages.ts` (`fetchSlackUserMessageHistory`) and orchestrated in `src/server/actions/slack/roster-enrich-from-messages.ts` (`buildSlackMessageEnrichmentForUser`). The fetch **merges two queries**: `from:<@USERID>` (their own posts; ascending sweep still pins oldest for join-date) and `<@USERID>` (messages that **@-mention** them — e.g. a founder’s welcome in a **group DM** that states their title). Role inference uses the merged transcript; join-date fallback uses **self-authored** hits only so an old unrelated @mention does not move tenure.

- **Role + Department (blank only):** the most recent ~50 messages are condensed (context tag + trimmed text) and sent to Claude via `claudePlainText` with the list of **departments already in use on the roster**. The model returns `{role, department, confidence}`. Only persisted when `confidence ≥ 0.35`. Department output is **canonicalized** back to an existing roster label (case-insensitive match) when possible, so the team doesn't grow near-duplicate labels like `Engineering` vs `Engineers`.
  - **DM-aware context tags.** Each message is prefixed with `[#channel-name]`, `[DM]`, `[group DM]`, or `[private group]` so the AI can distinguish channel signal (where the channel name itself is a strong topic hint) from 1:1 / group-DM content (where Slack's `channel.name` is a useless target-user ID and the AI must rely on content alone). Lines from **other authors** in the merged set are suffixed with `· teammate` in the condensed transcript so intros like *"Ana will be our Chief Of Staff"* are weighted correctly.
  - **Message-mix summary.** The user payload includes a short line (e.g. *"All 7 messages are DMs (no channel-name signal available — rely on content)."*) so Claude calibrates its confidence heuristic correctly for mostly-DM users.
  - **Minimum message count: 2.** Anything below that still short-circuits without calling the AI, since content-based inference becomes noise at 0-1 hits.
- **Join Date (blank only):** the **oldest self-authored** `from:<@USERID>` hit's `ts` is converted to `YYYY-MM-DD` (UTC) and saved (mention-query hits are ignored for tenure). Slack prunes search history on free plans and hides some private channels, so this is a best-effort fallback that runs only after `users.profile.get` `start_date` + custom ISO fields both came back empty.
- **Existing values are never overwritten.** If the row already has `role` set, the AI call is skipped entirely for that row. If it already has `joinDate`, the oldest-message math is skipped. Refresh simply no-ops for rows with both fields already filled in.
- **Scope failure ≠ user-visible error.** Missing `search:read` / `not_allowed_token_type` is logged through `SLACK_JOIN_DATE_DEBUG` and silently skipped so the rest of the refresh/import still writes.

## Slack channel picker (Goals)

The Slack column uses a **channel picker** that calls `conversations.list` (`types=public_channel,private_channel`). Token order: **`SLACK_CHANNEL_LIST_USER_TOKEN`** (optional user xoxp-) → else **`SLACK_BILLING_USER_TOKEN`** → else **`SLACK_BOT_USER_OAUTH_TOKEN`**. A **user** token lists non-archived channels **that user** can access (including private channels they’re in, e.g. workspace admin). The **bot** token only includes private channels the **bot** joined. **Search** filters by name, topic, or purpose.

When the goal’s **company** is known, **Relevant only** (on by default each time the panel opens) pre-filters to channels whose name, topic, or purpose contains the company **name** or **short name** (case-insensitive); turn it off to list every channel. Selecting stores **channel name** (`slackChannel`) and **channel ID** (`slackChannelId`). The **external-link** icon opens Slack via `https://slack.com/app_redirect?channel={id}`. Use user token scopes `channels:read` and `groups:read` (or `admin`) for listing; bot scopes `channels:read` / `groups:read` apply when listing falls back to the bot.

## Milestone Slack threads (Roadmap)

Each milestone may store **`slackUrl`** (HTTPS permalink to a channel message or thread). Parsing supports `…/archives/C…/p…` and `app.slack.com/.../thread/C…-ts` URLs (`parseSlackThreadUrl` in `src/lib/slack/threads.ts`, re-exported from `@/lib/slack`). Archive links that point at a **reply** in a thread include `?thread_ts=…` with the **root** parent message ts; the parser prefers that so thread fetch uses the correct anchor.

**Read + post** use the **user** token only (`slackUserTokenForThreads`: `SLACK_CHANNEL_LIST_USER_TOKEN` → else `SLACK_BILLING_USER_TOKEN`; no bot fallback) so the workspace member can call **`conversations.replies`** and **`chat.postMessage`** in channels they belong to without inviting the bot. Required **User Token Scopes:** `channels:history`, `groups:history`, `chat:write`.

The UI (`MilestoneSlackThreadInline` beside the milestone name, plus `SlackThreadPopover`, `SlackPingDialog`) shows last-activity vs **24h** staleness, recent lines (with **Team** `slackHandle` + name + profile photo when the message author matches the roster), **deadline risk** and a **one-line thread summary** when the milestone has a **target date** — both come from a single **`assessMilestoneOnTimeLikelihood`** Claude call (thread pace vs calendar, owner **autonomy**, project **complexity**, optional roadmap context; JSON includes `threadSummaryLine` plus likelihood fields), a **Nudge deadline…** draft (`generateDeadlineNudgeMessage`, first-person deadline emphasis), and a **ping** flow that drafts a generic follow-up with **`generateThreadPingMessage`** and sends after confirm via **`pingSlackThread`**.

All three generators (`generateThreadPingMessage`, `generateDeadlineNudgeMessage`, `reviseSlackThreadPingMessage`) build an **Authorship** background block: they resolve the sender via `auth.test` on the user token (preferring the Team roster name) and receive the project owner's name as `assigneeName` (from **`SlackMilestoneThreadPopovers`** → **`SlackPingDialog`**), so Claude writes in first person as the signed-in user, never @-mentions or names itself, and addresses the assignee with their **`<@USER_ID>`** token when asking for an update or pushing on the deadline.

The optional **`summarizeSlackThread`** action remains on the server for other use but is not used by the thread popover.

Likelihood results are stored in memory and **localStorage** (90-day retention; `src/hooks/useMilestoneLikelihood.ts`, `src/lib/browserJsonCache.ts`). Payloads include **`schemaVersion: 2`** and LS content keys use the **`mlh:v2:`** prefix so older entries (e.g. before **`threadSummaryLine`**) are ignored and re-assessed. A new **Claude** assessment runs when there is **no** stored result yet, when the cache is **legacy** (before reply-count tagging or schema **v2**), or when **`replyCount`** from `fetchSlackThreadStatus` **differs** from the count at last assessment (new replies). Assessments are **queued** with limited concurrency (`src/lib/likelihoodAssessQueue.ts`) so large Roadmaps do not stampede the API.

**Auto-complete at 100%:** When a fresh assessment reports **`progressEstimate === 100`** and the milestone is still **Not Done**, `useAutoCompleteMilestoneAt100` (`src/hooks/useAutoCompleteMilestoneAt100.ts`) calls `updateMilestone(id, { status: "Done" })` once, keyed by `(milestoneId, replyCount)` so it never fires twice for the same reading. The attempt marker is persisted in localStorage under the **`ecc.autoMsDone.v1:`** prefix (90-day TTL), so if the user manually reverts the milestone back to Not Done the system won't immediately re-complete — it only retries after new Slack thread activity (different `replyCount`) triggers a fresh AI assessment that still reads 100%. The hook is invoked from both `MilestoneRow` (expanded roadmap) and `ProjectRow`'s next-pending milestone (collapsed), and a module-level dedupe set ensures only one `updateMilestone` call fires when both subscribe to the same milestone.

**Collapsed goal row (Next column):** When a **goal** is collapsed, the Roadmap shows a compact strip after the goal **Slack** column: a **freshness dot + `Xd ago` label** (most-recent `lastReplyAt` across every next-pending milestone thread under the goal — emerald when at least one thread moved in the last 24h, amber pulse when every thread is stale, zinc ping while the thread-status cache is still hydrating), rollup **on-time %** and **AI confidence** (progress × on-time across projects), an **owner avatar stack** anchored to the summary (distinct project owners, sorted by **autonomyScore** desc then name), plus a **one-line goal summary**. The strip is a clickable button (`GoalLikelihoodInline`) that opens the **Goal delivery popover** (see below). The freshness signal is computed by `readSlackThreadFreshness` in `src/lib/slackThreadFreshness.ts`, which reads the same `slackThreadStatusCache` entries the milestone rows hydrate — so no additional Slack API calls are issued at the goal level; `useGoalLikelihoodRollup` polls the cache at 1s intervals until every thread's status has landed. The rollup is **deterministic**: for each child project it uses the **next pending** milestone with a valid **target date** and **Slack thread URL**, and reads the existing milestone likelihood from the shared cache (`readCachedMilestoneLikelihood` in `useMilestoneLikelihood.ts`; weighted by project **complexity**, floored by the weakest milestone, worst **riskLevel**). It does **not** call Slack again at the goal level. The summary sentence is a separate **`assessGoalOneLiner`** server action (`src/server/actions/ai/goal-one-liner.ts`) that takes those per-milestone **`threadSummaryLine`** values and rollup scores; results cache in localStorage under the **`gol:v1:`** prefix (90-day TTL, `useGoalOneLiner.ts`). If no project has an assessable next milestone (dated + Slack URL), the strip is omitted.

### Goal delivery popover (`GoalSlackPopover`)

Clicking the collapsed goal strip opens a portal popover anchored to the inline control, reusing the same `SlackThreadSpotlightBackdrop` dim layer as the milestone popover. The left panel shows the rollup **on-time %** (big number, risk-colored), **AI confidence** bar, per-project coverage count (`cached/total`), distinct **owner count**, the goal **one-liner summary**, and a **Reasoning** list that calls out the two worst-risk projects (risk desc, likelihood asc). The right panel is a **per-project drill-down**: one mini-card per project with its next pending milestone, owner avatar, **on-time %** + **progress %** pills, the per-project `threadSummaryLine`, and a direct **View thread →** link to the milestone's Slack URL. Rows are sorted worst-first (risk desc, likelihood asc). All data comes from the already-cached `goalLikelihoodRollup.projectSummaries`; the popover does **not** fetch channel history.

The popover's **Actions** menu opens the **`SlackChannelMessageDialog`** composer in **ping** / **nudge** / **reply** modes when the goal has a `slackChannelId`. **Back** (and Esc, backdrop click, or the header dismiss control) closes the composer with `dismiss` so the parent can reopen **Goal delivery**; a successful **Post to Slack** closes with `posted` and leaves the popover closed. In **reply** mode the composer opens in **Edit** with an empty textarea (optional **Draft with AI** via `draftGoalChannelMessage`); **ping** / **nudge** auto-draft from goal context. Sending calls **`postGoalChannelMessage(goalId, text)`** (`src/server/actions/slack/goal-channel-post.ts`). The message is a top-level channel post; the permalink is not stored on the goal.

Message text normalizes Slack **mrkdwn** (user/channel mentions, links, `:emoji:` shortcodes via a small Slack-first map plus **`node-emoji`** / emojilib so standard names like `:smile:` and Slack-specific names like `:film_frames:` render as Unicode in previews). Server actions live under `src/server/actions/slack/` (barrel `src/server/actions/slack/index.ts`); thread **status** uses a **one-hour** cache (popover reuses a fresh entry; **force** refresh after a successful ping). Keys: thread URL and roster ids when hints are used (`src/lib/slackThreadStatusCache.ts`). The milestone **Slack thread** popover lists the **last five** thread replies for preview.
