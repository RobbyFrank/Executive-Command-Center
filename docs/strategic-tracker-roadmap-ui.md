# Roadmap UI: editing, filters, URLs

MLabs-internal. For data model and storage, see [strategic-tracker-data-model.md](strategic-tracker-data-model.md). For Slack, see [strategic-tracker-slack.md](strategic-tracker-slack.md). For AI routes, see [strategic-tracker-ai-auth.md](strategic-tracker-ai-auth.md).

## Project and goal workflow

**Project** workflow statuses (`ProjectStatusEnum` in `tracker.ts`) are **Idea**, **Pending**, **In Progress**, **Stuck**, **Blocked**, **For Review**, and **Done** — distinct from **goal** statuses (`GoalStatusEnum`). **Blocked** is not chosen in the Status dropdown: it appears when another project is set as **blocked by** and that project’s milestones are not all done (the stored workflow status is unchanged). Legacy project status strings in JSON are normalized on load (e.g. `Not Started` → `Pending`; legacy `Blocked` in old files loads as **Blocked**).

When a milestone’s **`slackUrl`** is saved as a valid `https` permalink and that milestone previously had no valid URL, if the parent **project** is still **Idea** or **Pending**, the server sets the project to **In Progress** (`createMilestone` / `updateMilestone` in `src/server/actions/tracker.ts`).

**Under each goal**, primary projects and **mirrors** of that goal are ordered together so **priority** (P0 first) applies, but any project whose **`blockedByProjectId`** points at another project in that same list is shown **directly under** that blocker (dependents of the same blocker are sorted by priority; chains nest depth-first).

The Roadmap filter bar includes a **Show completed** checkbox: when unchecked, **Done** projects are hidden and goals that would have no visible projects are omitted.

## Scores, priority, owners

**Delay cost** (on goals; column label — same width as other roadmap data columns) and **complexity** (on projects) are edited with dropdowns 1–5 (`costOfDelay` on goals; `complexityScore` on projects — **complexity** is on the **collapsed** project bar, immediately before **Confidence**). Priority, Delay cost / Complexity, and Confidence share **`ROADMAP_DATA_COL_CLASS`** (`w-28`). Goal **Slack** uses **`ROADMAP_GOAL_SLACK_COL_CLASS`** (`w-max`) so the channel name can grow; the picker shows a **pencil** inline after the name on hover (or when the edit control is focused). **Owner** is avatar-only and uses **`ROADMAP_OWNER_COL_CLASS`** (`w-12`) so the column is not as wide as other data cells. **`ROADMAP_GRID_GAP_CLASS`** (`gap-2`) sits between grid cells. **Priority** is edited on goals and projects separately from those scores. The Roadmap shows **Urgent / High / Normal / Low** with colored **flag** icons; JSON still stores **`P0`–`P3`**.

**Confidence** is **auto-calculated** and shown on both goal and **collapsed** project rows. The **goal** title column is **`calc(360px + 2.5rem)`** wide and the **project** title column stays **360px**, so **Owner** and all following columns line up across goal vs project rows while project rows stay indented (`pl-14` vs `pl-4` on the grid). Expanded **milestone** rows sit in a nested strip under the project; milestone names use the same **`ROADMAP_ENTITY_TITLE_DISPLAY_CLASS`** (**13px** semibold) as goal and project titles (`ROADMAP_MILESTONE_GRID_PADDING_CLASS` — `pl-24` vs `pl-14` on projects and `pl-4` on goals — / `ROADMAP_GOAL_GRID_PADDING_CLASS` in `src/lib/tracker-roadmap-columns.ts`). Scoring (score **0–5** → **0%–100%**): per project, **no owner or owner not on Team → 0**; otherwise `clamp(1,5, ownerAutonomy - complexity + 3)`; per goal, it is the **plain average** of its child project confidence scores, rounded to the 0–5 band. If there are no projects, the goal shows **0/0%** until projects exist. Hover or focus the cell for a small panel (not a browser tooltip): **project** rows include a short rationale; **goal** rows show the aggregate score and a one-line note that it is an average (no per-project list). Legacy `confidenceScore` on goals in JSON is not edited on the Roadmap grid. Legacy `impactScore` on goals may still be present in JSON but is not edited in the app.

On **goal** rows, **Due date** and **Progress** are read-only rollups: **Due date** is the **latest** (maximum calendar) milestone `targetDate` among all projects under the goal (`goalLatestMilestoneDueDateYmd` in `src/lib/goal-milestone-aggregates.ts`). **Progress** uses the same **Done / total milestone** rule as projects (`milestoneProgressPercent` on all milestones in the goal’s projects). The **Next milestone** column (`ROADMAP_NEXT_MILESTONE_COL_CLASS`) is normally an empty spacer on **expanded** goal rows so project **Next milestone** strips align; when the **goal is collapsed** and at least one child project has a dated next pending milestone with a **Slack thread URL**, it shows **`GoalLikelihoodInline`**: a **freshness dot + relative "Xd ago" label** (most-recent `lastReplyAt` across all threads rolled up into the goal), rollup on-time %, **AI confidence** (distinct from the **Confidence** column), owner avatar stack, and a one-line summary (`useGoalLikelihoodRollup` / `assessGoalOneLiner` — see [strategic-tracker-slack.md](strategic-tracker-slack.md)). Freshness reads synchronously from the shared thread-status cache (no extra Slack calls at the goal level) and uses the same three-state dot as `MilestoneSlackThreadInline`: **emerald** when at least one thread moved in the last 24h, **amber pulse** when every thread is older than 24h, **zinc ping** while the cache is still hydrating. The summary text uses the **same font / color / size** as the `MilestoneSlackThreadInline` body (`text-[11px] font-normal text-zinc-500 → zinc-400 on hover`) so both inline previews read as the same text style. On **collapsed** goal/project rows the Next-milestone column **grows into the trailing flex-1 spacer** (`!w-auto !grow min-w-[36rem]`) so the one-line summary / thread preview gets more horizontal room than the baseline 36rem slot; on narrower viewports the 36rem minimum keeps grid alignment with the sticky headers. Equivalent caps on the inline previews are `max-w-[48rem]` (project next-milestone strip) and `max-w-[min(54rem,calc(100% - max(4.5rem,3vw)))]` (milestone-row absolute strip at the goal Confidence column).

**Owner** on goals and projects in the Roadmap grid is **avatar-only** (profile photo or initials in a circle); hover for full name. **Unassigned** shows a quiet dashed **person +** icon (no amber “Unassigned” label). The goal **DRI** column (directly responsible individual for the goal outcome) only lists **founders** and people with **autonomy 4 or 5**; the picker shows a short note and the server rejects other assignments. Project **owners** still use the full team roster. Other surfaces that use `OwnerPickerCell` without the compact grid mode may still show name text. Project **`type`** (Engineering, Product, etc.) remains in JSON but is not edited on the Roadmap grid.

## Collapsed project bar and goal rows

On the **collapsed** project bar, **workflow status** is a small circular icon before the project name (same colors as status); clicking it opens the usual **Status** overlay dropdown (there is no separate Status column). Columns after **Priority** are **Complexity** (1–5 band, shown as three ascending **signal bars** + short label — aligned under the goal **Delay cost** column), **Confidence** (aligned under the goal **Confidence** column; the sticky **Goals** label row already names **Confidence**, so project rows do not repeat that word in a second label row), **Due date** (read-only: same calendar date as the **last milestone** that has a **target date**; shown as `targetDate` on the project in the UI after hierarchy load; urgency tint vs local end-of-day: **red** overdue, **orange** within 24h of that moment, **yellow** within 48h — see `getProjectDueDateUrgency` in `relativeCalendarDate.ts`), **Progress**, then **Next milestone** (first milestone in list order not **Done**; a compact **D**/**W**/**M**/**Y** horizon from its milestone **target date** when set, otherwise **—** — hover for full relative wording, calendar date, and **Milestone *k*/*n***; when the project row is **collapsed** and that milestone has a **Slack thread URL**, a compact last-reply preview appears beside the name; when it has **no** thread URL yet, a **Start Slack thread** control (`StartSlackThreadChip` — Slack + **+** icon, same on collapsed project and expanded milestone rows) opens the same menu as the milestone Slack affordance — **Draft a new Slack thread with AI…** or **Attach existing Slack thread URL…** (the latter expands the project and focuses the URL field on the next milestone row); when the project is **expanded**, the project row no longer shows that strip — the **next milestone** row shows **`MilestoneSlackThreadInline`** aligned with the **goal** row **Confidence** column (`TRACKER_ROADMAP_MILESTONE_SLACK_INLINE_AT_GOAL_CONFIDENCE_LEFT`); when there is **no** thread URL yet, the same **Start Slack thread** icon control uses that horizontal alignment.

Each **project** under a goal is a **card** (`ProjectRow`: border, left accent, background) joined to the goal by a small **tree** (vertical spine at the goal-chevron column + a short horizontal stub per project) drawn in `GoalSection`. Project cards are indented by **`ROADMAP_PROJECT_CARD_INDENT_PX`** (**56px** = `pl-14`) from each goal row's flex start so they begin where the stub ends; the project row's own padding is **0** (`ROADMAP_PROJECT_GRID_PADDING_CLASS = pl-0 pr-3`) so Owner and following columns still line up with goal rows. Expanded **milestone** lists use a light **shelf** only — **`ROADMAP_MILESTONE_LIST_SHELF_CLASS`** (`border-t` + subtle tint in `tracker-roadmap-columns.ts`) — so milestones are not wrapped in a second band that matched the old goal-level project strip. Milestone rows use **`ROADMAP_MILESTONE_GRID_PADDING_CLASS`** (`pl-24 pr-3`) — one more **2.5rem** step-in than project rows so Project→Milestone spacing matches Goal→Project. **`ROADMAP_MILESTONE_NEXT_CHIP_SLOT_CLASS`** reserves a fixed **Next** chip column on every milestone row (empty when not the next pending milestone). With a linked Slack thread, **`MilestoneSlackThreadInline`** is absolutely positioned at **`TRACKER_ROADMAP_MILESTONE_SLACK_INLINE_AT_GOAL_CONFIDENCE_LEFT`** so its left edge lines up with the **goal** row **Confidence** column (the calc subtracts `ROADMAP_PROJECT_CARD_INDENT_PX` because milestone rows live inside an indented project card); **`ROADMAP_MILESTONE_TITLE_MAX_WHEN_SLACK_THREAD_STRIP_CLASS`** caps the milestone **title** cell (after the date + Next chip columns) so long names truncate before that strip — it subtracts the same leading in-row width (**17.75rem** from checkbox through Next slot) from the slack anchor, not merely the project title column width.

**Description** and **Done when** on projects (`description` / `definitionOfDone`) are edited from the **info icon** beside the project name (floating panel; **click a field** to edit inline). Plan dates in **milestones**; the project due date follows the last dated milestone.

On each **goal** row, columns after **Priority** are **delay cost**, **Confidence**, rollup **Due date**, **Progress**, **Slack** (channel picker — after Progress), then the **Next milestone** column (spacer when the goal is expanded; **collapsed-goal delivery strip** when applicable — see previous paragraph). **Goal description** (`measurableTarget`), **Why** (`whyItMatters`), and **Current** (`currentValue`) are edited from the **info icon** beside the goal title (same panel), not in the sticky column header row.

## Inline editing and context menus

**Roadmap** uses **inline editing** for goals, projects, and milestones (default toolbar: tree expansion **Goals only**; **Focus** on/off is persisted in the browser). **Right-click** (context menu) on a **company** header, **goal** row, **project** row, or **milestone** row—or click the row **⋯** (same menu)—for quick actions: e.g. add goal/project/milestone, **rename** (goal title or project name — opens the inline title field), **Update with AI…** (goals and projects — opens `POST /api/ai-update`, see [strategic-tracker-ai-auth.md](strategic-tracker-ai-auth.md)), **Review notes…** (goals and projects — opens the log popover), **Discuss in chat** (goals, projects, milestones — opens the Assistant with that entity tagged), **Mirror to goal…**, **Set blocked by…** (pick any other project; **`blockedByProjectId`** in JSON), **Clear blocked by**, expand or collapse, set **At risk** / **Spotlight** / clear signal, and **delete** (with confirmation when destructive).

While a blocker’s milestones are incomplete (or it has no milestones), the dependent project shows a **Blocked** badge; the badge hides when the blocker reaches 100% milestone progress. Hierarchy also exposes **`isBlocked`** and **`blockedByProjectName`** on project rows for UI.

**Add project** and **Add goal** (including the empty-company placeholder row) use a single control that opens an anchored **context menu**—same pattern as the next-milestone **Slack +** chip: **Draft … with AI…** (opens `AiCreateDialog`) or **Add blank …** (creates the default entity). They sit at the bottom of each goal’s project list or alone when there are no projects; **Add goal** still scopes new goals to the **company** (`companyId`). **Companies** (name, short name, revenue, logo, add/remove) are managed on the **Companies** page (`/companies`); company rows on Roadmap are read-only headers. A **company** can only be deleted when it has **no goals** (remove or move goals first). A **goal** can only be deleted when it has **no projects** (delete projects first). A **person** can only be deleted when they are not a **goal owner**, **project owner**, or **assignee** on any project.

## Sticky headers

The **Goal** column label row (`GoalsColumnHeaders` in `TrackerColumnHeaders.tsx`) is **sticky** below the measured height of the sticky title/filter toolbar. The **company** header row sticks for the **entire** company block (until that section scrolls away), with **goal** column labels and each **goal** header row stacked beneath in order (`top` + `z-index`). There is no separate sticky **Project** column label row under each goal (project cards align visually under the goal grid). Individual project title rows are not sticky so the company bar stays the section-level anchor.

## Search and filters

Use the **search** field at the top of Roadmap to filter by substring across company names and short names, goal text and fields, project names and metadata (including owner and assignee **names** resolved from **Team**), and milestone names and dates.

**Companies** and **Owner** multi-select filters narrow the tree (selected companies only; then goals/projects owned by any selected **person**, anyone in a selected **department**, anyone at a selected **autonomy** level (1–5, from **Team**), or anyone matching a selected **employment** type on **Team** — combined with OR across selected tokens). **Priority** (P0–P3) and **delivery status** (project workflow `status` values) are separate multi-selects.

**Owner** cells show an **amber ring** (profile photo) or **dot** (name-only row) when the assignee’s autonomy is 1–2 (founders excluded). On Roadmap (avatar-only owner column for goals **and** projects), each owner avatar also carries a small **autonomy badge** slightly outside the photo’s bottom-right (`OwnerAutonomyBadge` in `src/components/tracker/OwnerAutonomyBadge.tsx`): a **♛** for **founders**, **?** for autonomy **0** (not assessed), and **1–5** otherwise — white glyphs on saturated amber / zinc / emerald fills (same bands as the `A{n}` pills in the owner picker). Hover the badge for the level’s long name (e.g. "Autonomy 4. High ownership").

The **signals** multi-select shows rows where **any** chosen signal applies. **Manually flagged:** **Flagged at risk** (`atRisk` flag on goal or project), **Spotlighted** (`spotlight` flag — win or positive momentum). **Auto-detected:** **Unassigned** (no DRI on the goal or owner on the project), **Stuck in progress** (project In Progress but no milestones completed yet — the "zombie" check), **Needs kickoff** (goal delay cost ≥4 and goal status not In Progress — distinct from calendar-based overdue, which lives in the Dates filter).

The **due date** multi-select filters **projects** by `targetDate` using **cumulative** horizons. Filter order: company → owner → priority → status enum → signals → due date → search.

## URL query parameters

**Roadmap** (`/`) supports the same query parameters the UI uses: comma-separated `companies` (ids), `owners` (ids or owner-filter tokens), `tags` (signal ids: `at_risk`, `spotlight`, `unassigned`, `zombie`, `stalled`, `new_hire_pilot`), `priorities` (`P0`…`P3`), `delivery` (project status enum values), `due` (due-date bucket ids), `q` (search text), plus `focusGoal` / `focusProject` for a single expanded project.

**Review notes…** in the row menu opens the **review log** popover (append notes, read history). Dedicated **Summary**, **Matrix**, and full-page **Review** experiences are not part of this build (may be added later).

Goals within a company and projects within each goal are sorted by **priority** (P0 first).

## Persistence

Changes persist through server actions: each mutation reads the document, applies updates, bumps `revision`, and writes with an atomic compare-and-swap in Redis (retries on conflict; stale concurrent edits surface an error instead of silent data loss).
